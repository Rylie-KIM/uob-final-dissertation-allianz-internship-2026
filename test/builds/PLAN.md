### Brief Statement 1
> "A self-fulfilling prophecy in machine learning occurs when a model's predictions influence the outcome in a way that reinforces the model's original prediction, creating a feedback loop."

**Answered by:** Pillar 1 — Build 01 (SFP Simulation)

The loop is formalised as a closed cycle:

```
Model Prediction → Investigation Decision → Observed Fraud Label
       ↑__________________________________________________|
                (biased label feeds back into retraining)
```

Build 01 simulates this on synthetic insurance claims data. A model trained on investigation-selected labels will systematically overestimate precision in investigated claim types while being blind to fraud in uninvestigated segments — even as its reported metrics appear stable.

**Pre-experiment hypotheses:**
- H₀: A model retrained on labels generated by its own previous predictions achieves the same true fraud recall as a model trained on randomly sampled labels.
- H₁: The retrained model shows inflated precision but lower true recall — confident about a shrinking slice of reality.

**Experimental design:**
1. Generate synthetic claims with known ground-truth fraud rate
2. Train Model v1 on random labels → use scores to select which claims to investigate → generate biased observed labels
3. Retrain Model v2 on biased labels → compare v1 vs v2 true recall against ground truth
4. Repeat for 3–5 loop iterations to show degradation trajectory
5. Quantify: loop-induced recall gap = true_recall(v1) − true_recall(vN)

### Brief Statement 2
> "This can lead to biased or inaccurate results, especially in systems that interact with human behaviour or decision-making processes."

**Answered by:** Pillar 1 (mechanism explanation) + Pillar 3 (quantifying bias via unbiased evaluation)

#### Why Human-Behaviour and Decision-Making Systems Are Uniquely Vulnerable

Two distinct mechanisms explain why SFP loops are especially dangerous in these settings:

**(A) Human behaviour is adaptive — predictions change the system being predicted**

In a purely physical system, a prediction does not alter the physical world: a weather forecast does not cause rain. But when a human agent — an insurance investigator, a loan officer, a content moderator — receives a model score, they *act on it*. That action determines which outcomes get observed and subsequently labelled as ground truth.

The model's output literally reshapes the data distribution it will be trained on next. This is the core of **Performative Prediction** (Perdomo et al., 2020): the model changes the very distribution it is trying to learn. No physical sensor system has this property.

**(B) Decision-making processes create selection bias in labels**

When a decision-making system uses model scores to choose which items to examine (which claims to investigate, which applications to review), only selected items receive ground-truth labels. Items the model scores low are never examined — so any fraud, failure, or error in those items is permanently unobserved.

The resulting label set is not a random sample of reality; it is a *model-curated* sample. Standard evaluation metrics computed on this sample appear strong because the model's blind spots are simply never measured.

**Together:** mechanism (A) means that human responsiveness continuously amplifies the model's initial biases into the next training cycle. Mechanism (B) means the corrupted label pool never reveals what the model is missing. Each training cycle reinforces the previous one. In contrast, a turbine-failure prediction model trained on sensor data does not exhibit (A) because sensor readings are not influenced by the model's outputs.

**Insurance-specific example (motor fraud):**
Fraud model assigns high scores to whiplash motor claims → investigators focus investigations on whiplash → fraud is found and labelled → next training data over-represents whiplash fraud → model becomes even more confident about whiplash → staged arson claims, organised ghost-broker fraud, and other types are systematically missed → loop tightens across each model version update.

### Brief Statement 3
> "Identifying these loops can be tricky."

**Answered by:** Pillar 2 — Build 02 (Loop Detection Framework)

**Why identification is hard:** The loop degrades performance *silently*. Standard metrics (accuracy, precision, AUC-ROC) computed on the same biased labels the model was trained on appear stable or even improving, even as true recall against real fraud deteriorates. There is no natural alarm signal visible from inside the loop.

**Why multiple model versions help:** If v1 predictions influenced which claims v2 was trained on, comparing v1 and v2 score distributions, label rates, and investigation patterns reveals statistical fingerprints of the loop — even without access to ground-truth fraud labels.

**Detection signals used in Build 02:**
1. Temporal shift in prediction score distributions across model versions
2. Correlation between model score and investigation rate (strong correlation = loop signal)
3. Label rate in high-score vs low-score segments across versions
4. Counterfactual label rate in any random-sample holdout that exists

---

### Brief Statement 4
> "Can we create a framework to identify these loops and suggest for improvements in the future?"

**Answered by:** Pillar 2 — the `SFPDetector` Python module (Build 02)

**Framework — 4-step diagnostic pipeline:**

| Step | Method | Output |
|------|--------|--------|
| 1. Temporal prediction audit | KS-test on score distributions across model versions | p-value for distributional drift |
| 2. Label-generation mechanism test | Logistic regression: P(investigated \| score, features) | Propensity model + loop risk score |
| 3. Action-outcome confounding test | Partial correlation: score vs label, controlling for investigation propensity | Residual association after deconfounding |
| 4. Intervention recommendation | Ranked list based on loop risk score | Which mitigation to apply first |

**Suggested improvements output:** The framework recommends one of: (a) epsilon-greedy random investigation injection, (b) IPW-debiased retraining, (c) RDD analysis at score threshold, or (d) temporal holdout reset — ranked by estimated loop severity.

### Brief Statement 5
> "We can start from unbiased performance evaluation and intervention analysis to assess model driven impact."

**Answered by:** Pillar 3 — Build 03 (Unbiased Evaluation) + Pillar 4 — Build 04 (Intervention Analysis)

#### Pillar 3 — Unbiased Performance Evaluation

**Why standard evaluation fails:** A train/test split on biased labels is still biased — the test set was generated by the same selection mechanism as the training set. The model is evaluated only where it already looked.

**Methods implemented:**
- **Temporal holdout:** evaluate on claims from a later time period than the training window, before the next model update could have influenced those labels
- **IPS-corrected metrics:** each label observation is weighted by 1/P(investigated | features), correcting for the over-representation of high-score claims in the label pool
- **Comparison output:** biased_recall vs IPS_corrected_recall — the gap measures the loop-induced optimism in standard evaluation

**Pre-experiment hypotheses (Pillar 3):**
- H₀: IPS-corrected recall equals uncorrected recall → investigation propensity does not distort the label distribution.
- H₁: IPS-corrected recall is significantly lower → the model's apparent performance is partially a selection artefact.

#### Pillar 4 — Intervention Analysis

**Key question:** When Allianz updated from model v1 to v2 (a real intervention), did the transition change the fraud label distribution in ways attributable to the model change rather than underlying fraud rate change?

**Methods:**
- **DiD (Difference-in-Differences):** compare fraud label rates before/after model version update, in claim segments differently affected by the score change vs unaffected segments
- **RDD (Regression Discontinuity Design):** if the model uses a hard investigation threshold, claims just above and just below the threshold are near-identical in all respects except investigation status — this gives a clean causal estimate of investigation effect on labelling

**Pre-experiment hypotheses (Pillar 4):**
- H₀: Fraud label rates changed equivalently across all claim segments after each model version update.
- H₁: Fraud label rates changed more in segments the new model scores high — the model version change caused differential outcome labelling, not a real change in underlying fraud.

### Brief Statement 6
> "Then potentially exploring randomisation strategies and causal inference techniques for mitigation."

**Answered by:** Pillar 5 — Build 05 (Randomisation) + Pillar 6 — Build 06 (Causal Inference)

#### Pillar 5 — Randomisation Strategies

**Core idea:** Deliberately inject random investigations regardless of model score. This creates unbiased label observations in segments the model would otherwise ignore, breaking the feedback loop.

**Policy comparison:**
- Policy A: pure model-driven (ε = 0) — investigate top K% by score only
- Policy B: epsilon-greedy (ε = 0.1) — 90% model-driven + 10% random
- Policy C: epsilon-greedy (ε = 0.3) — 70% model-driven + 30% random

**Pre-experiment hypotheses (Pillar 5):**
- H₀: Adding random investigation (ε > 0) does not improve model calibration or true recall after retraining.
- H₁: Randomised labels enable the retrained model to detect fraud types previously invisible to it; true recall increases with ε at the cost of investigation efficiency.

**Experimental design:**
1. Run the SFP simulation from Build 01 under each policy for 5 loop iterations
2. At each iteration, measure: true recall (vs ground truth), precision, loop-induced drift
3. Plot degradation curve per policy — show at what ε the loop is effectively broken

#### Pillar 6 — Causal Inference Mitigation

**Core idea:** Use DoWhy to specify the causal DAG, then estimate the causal effect of model score on fraud label *after* blocking the backdoor path through investigation propensity.

**DAG:**
```
Model Score → Investigation Decision → Fraud Label (observed)
    |                                         ↑
    └─────────── Claim Features ──────────────┘
```

The path `Model Score → Investigation Decision → Fraud Label` is a selection mechanism, not causal evidence of fraud. We block it using Inverse Probability Weighting (IPW), then retrain on the debiased labels.

**Pre-experiment hypotheses (Pillar 6):**
- H₀: The causal effect of model score on fraud label equals the observed correlation after controlling for claim features.
- H₁: After adjusting for investigation propensity (blocking the selection path), the association between model score and fraud label is significantly weaker — confirming that the loop created spurious correlation.

## 브리프 문항 1
> "자기실현적 예언(SFP)은 모델의 예측이 결과에 영향을 미쳐 원래 예측을 강화하는 피드백 루프를 만들 때 발생한다."

**어떻게 답변하는가:** Pillar 1 — Build 01 (SFP 시뮬레이션)

루프 구조를 아래처럼 형식화한다:

```
모델 예측 → 조사 결정 → 관측된 사기 레이블
     ↑____________________________________________|
             (편향된 레이블이 재학습 입력으로 유입)
```

Build 01에서 합성 보험 청구 데이터로 이 루프를 시뮬레이션한다. 조사-선택된 레이블로 학습한 모델은 조사된 클레임 유형에서는 정밀도가 과대평가되고, 조사받지 못한 영역의 사기는 아예 감지하지 못하게 된다.

**실험 전 가설:**
- H₀: 이전 예측이 생성한 레이블로 재학습한 모델의 진짜 재현율(true recall)은 무작위 레이블로 학습한 모델과 동일하다.
- H₁: 재학습 모델은 정밀도는 높아 보이지만 진짜 재현율은 낮다 — 자신이 이미 본 영역에 대해서만 자신감을 갖는 구조.

**실험 설계:**
1. 알려진 실제 사기율이 있는 합성 청구 데이터 생성
2. Model v1을 무작위 레이블로 학습 → 스코어로 조사 대상 선정 → 편향된 레이블 생성
3. Model v2를 편향 레이블로 재학습 → v1 vs v2의 진짜 재현율 비교
4. 3~5번의 루프 반복으로 성능 열화 궤적 측정
5. 루프 유발 재현율 감소량 = true_recall(v1) − true_recall(vN) 정량화

## 브리프 문항 2
> "이는 특히 인간 행동 또는 의사결정 프로세스와 상호작용하는 시스템에서 편향되거나 부정확한 결과를 초래할 수 있다."

**어떻게 답변하는가:** Pillar 1 (메커니즘 설명) + Pillar 3 (편향 정량화)

### 왜 인간 행동·의사결정 시스템이 특히 취약한가

두 가지 메커니즘이 SFP 루프를 이 환경에서 특히 위험하게 만든다:

**(A) 인간 행동은 적응적이다 — 예측이 예측 대상 시스템 자체를 변화시킨다**

물리 시스템에서는 예측이 물리 세계를 바꾸지 않는다 (날씨 예보가 비를 만들지 않는다). 그러나 보험 조사관, 대출 담당자, 콘텐츠 검토자처럼 인간 행위자가 모델 스코어를 받으면 그에 따라 행동한다. 그 행동이 어떤 결과가 관측되고 라벨링되는지를 결정한다. 모델의 출력이 다음 학습에 사용될 데이터 분포 자체를 바꾸는 것이다. 이것이 **Performative Prediction** (Perdomo et al., 2020)의 핵심이다.

**(B) 의사결정 프로세스는 레이블에 선택 편향을 만든다**

모델 스코어를 기반으로 조사할 항목을 선택하는 시스템에서는, 선택된 항목만 실제 레이블을 받는다. 스코어가 낮아서 선택되지 않은 항목의 사기·이상은 영원히 관측되지 않는다. 이렇게 만들어진 레이블셋은 현실의 무작위 표본이 아니라 **모델이 선별한 표본**이다. 이 표본으로 계산한 평가 지표는 모델의 사각지대가 측정에서 제외되었기 때문에 좋아 보인다.

**두 메커니즘이 결합하면:** (A)로 인해 인간 행동이 모델 편향을 다음 학습 사이클로 증폭시키고, (B)로 인해 부패한 레이블 풀이 무엇이 누락되었는지 드러내지 않는다. 물리 센서 기반 시스템 (예: 터빈 고장 예측)에는 (A)가 존재하지 않는다 — 센서 측정값은 모델 출력에 영향받지 않기 때문이다.

**보험 구체 사례 (자동차 보험 사기):**
사기 모델이 편타 손상 청구에 높은 스코어 부여 → 조사관이 편타 손상 청구 집중 조사 → 사기 발견 및 레이블링 → 다음 학습 데이터가 편타 손상 사기로 과대대표 → 모델이 편타 손상에 더욱 집중 → 무대 방화, 브로커 사기 등 다른 유형 사기는 체계적으로 미탐지 → 모델 버전이 업데이트될수록 루프 심화.

## 브리프 문항 3
> "이러한 루프를 식별하는 것은 까다로울 수 있다."

**어떻게 답변하는가:** Pillar 2 — Build 02 (루프 탐지 프레임워크)

**왜 탐지가 어려운가:** 루프는 성능을 *조용히* 저하시킨다. 편향된 레이블로 계산한 정확도·정밀도·AUC는 안정적이거나 개선되는 것처럼 보인다. 진짜 재현율이 떨어지고 있어도 내부에서는 경보가 울리지 않는다.

**여러 모델 버전이 도움이 되는 이유:** v1 예측이 v2 학습 데이터에 영향을 미쳤다면, v1과 v2의 스코어 분포·레이블 비율·조사 패턴을 비교하면 루프의 통계적 흔적이 드러난다 — 실제 사기 레이블 없이도.

## 브리프 문항 5
> "편향 없는 성능 평가와 모델 주도 영향을 평가하기 위한 개입 분석에서 시작할 수 있다."

**어떻게 답변하는가:** Pillar 3 — Build 03 (비편향 평가) + Pillar 4 — Build 04 (개입 분석)

**Pillar 3 — 비편향 성능 평가:**
편향된 레이블에 대한 훈련/테스트 분할은 여전히 편향되어 있다. 구현하는 방법:
- **시간적 홀드아웃:** 학습 기간보다 이후 청구를 테스트 세트로 사용
- **IPS 가중 지표:** 각 레이블 관측값에 1/P(investigated | features) 가중치 적용해 선택 편향 보정
- **비교 결과:** biased_recall vs IPS_corrected_recall — 그 차이가 루프 유발 낙관주의를 정량화

**Pillar 3 가설:**
- H₀: IPS 보정 재현율 = 미보정 재현율 → 조사 성향이 레이블 분포를 왜곡하지 않음.
- H₁: IPS 보정 재현율이 유의미하게 낮음 → 모델의 겉보기 성능이 선택 효과 산물임.

**Pillar 4 — 개입 분석:**
모델 버전 전환(v1→v2)을 자연 실험으로 활용. DiD로 "모델 업데이트 후 사기 레이블 분포 변화가 실제 사기율 변화보다 모델 스코어 변화와 더 일치하는가?"를 검정.

**Pillar 4 가설:**
- H₀: 사기 레이블 비율이 모델 버전 업데이트 후 모든 클레임 세그먼트에서 동등하게 변화.
- H₁: 새 모델이 높게 스코어링하는 세그먼트에서 사기 레이블 비율이 더 크게 변화 — 실제 사기 변화가 아닌 모델 버전 변화가 원인.

## 브리프 문항 6
> "무작위화 전략과 인과 추론 기법으로 완화를 탐색할 수 있다."

**어떻게 답변하는가:** Pillar 5 — Build 05 (무작위화) + Pillar 6 — Build 06 (인과 추론)

**Pillar 5 — 무작위화 전략:**
엡실론-탐욕 정책: 확률 ε로 모델 스코어와 무관하게 무작위 조사 배정. 편향되지 않은 레이블 관측을 생성하고 피드백 루프를 차단한다.

**Pillar 5 정책 비교:**
- Policy A: 순수 모델 주도 (ε=0) — 상위 K%만 조사
- Policy B: 엡실론-탐욕 (ε=0.1) — 90% 모델 주도 + 10% 무작위
- Policy C: 엡실론-탐욕 (ε=0.3) — 70% 모델 주도 + 30% 무작위

**Pillar 5 가설:**
- H₀: 무작위 조사(ε > 0) 추가가 재학습 후 모델 교정 또는 진짜 재현율을 개선하지 않는다.
- H₁: 무작위 레이블이 모델이 이전에 보지 못한 사기 유형 탐지를 가능하게 한다; ε 증가에 따라 진짜 재현율이 상승한다.

**Pillar 6 — 인과 추론 완화:**
DoWhy로 인과 DAG를 명시한 후, 조사 성향을 통제한 뒤 모델 스코어가 사기 레이블에 미치는 인과 효과를 추정. IPW 디바이어싱된 레이블로 수정 모델 재학습.

**Pillar 6 가설:**
- H₀: 모델 스코어의 사기 레이블에 대한 인과 효과 = 클레임 특성 통제 후 관측 상관관계.
- H₁: 조사 성향(선택 경로)을 차단한 후 스코어-레이블 연관성이 유의미하게 약해짐 → 루프가 허위 상관을 만들었음을 확인.
